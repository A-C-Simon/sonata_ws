# Вокселизованный кэш: процедура после map_from_scans

**Вариант 1 (реализованный пайплайн):** минимум места, один проход map_from_scans. Сначала `map_from_scans --save_only_map_world` → одна карта на seq (`map_world.npz`). Затем прекомпут с `--backend torch` (и при желании `--max_map_points`): один раз считаем карту, прекомпут переводит её в СК кадров и пишет кэш. Скрипты `run_voxelized_cache_semantickitti.py` и `run_lidar_to_sonata.py` делают именно это.

**Быстрый запуск:** из корня репозитория:
```bash
python scripts/run_voxelized_cache_semantickitti.py --data_path /path/to/SemanticKITTI/dataset
```
Если `map_world.npz` уже есть: `--skip_map`. Чтобы можно было закрыть терминал, запускайте через обёртку (логи в `logs/`):
```bash
./scripts/run_voxelized_cache_background.sh --data_path /path/to/dataset [--skip_map] [--skip_existing] [--verbose]
```
В конце скрипт выведет команду для обучения с `--voxelized_cache_dir`.

---

## Зачем это нужно

- **map_from_scans** строит одну карту на последовательность (`ground_truth/XX/map_world.npz`) в мировой СК. Этого достаточно для обучения, но при каждом батче датасет делает тяжёлую работу: переводит всю карту в систему координат текущего кадра и вокселизует — из‑за этого CPU перегружен, GPU простаивает.
- **Вокселизованный кэш** — это шаг **после** map_from_scans: один раз по каждому кадру (pose + seq) считаем «карту в координатах этой позы» + вокселизацию и сохраняем маленький файл. При обучении только читаем эти файлы — без пересчёта карты и вокселизации. Обучение ускоряется, GPU загружен выше.

---

## Что предполагается на входе (после map_from_scans)

Структура датасета (SemanticKITTI):

- `{data_path}/sequences/XX/` — velodyne (`.bin`), labels (`.label`), `poses.txt`, `calib.txt`.
- `{data_path}/ground_truth/XX/map_world.npz` — одна карта на последовательность (результат map_from_scans). Внутри: массив точек в мировой СК (например ключ `points`).

Никаких отдельных «карт на каждый кадр» на этом шаге не требуется — только одна карта на seq.

---

## Алгоритм процедуры (пошагово)

Выполняется **один раз** после map_from_scans, по всем нужным последовательностям и кадрам.

### Входные параметры

- `data_path` — корень датасета (там же где `sequences/` и `ground_truth/`).
- `output_dir` — куда класть кэш (например `{data_path}/voxelized_cache`).
- `voxel_size` — шаг воксельной сетки (например 0.05), тот же, что при обучении.
- `max_points` — лимит точек на partial и на complete после субсэмплинга (например 20000).
- Список `sequences` — номера seq (например 00, 01, …, 10 для train).

### Для каждой последовательности XX

1. Загрузить общую карту один раз:  
   `map_world = np.load("{data_path}/ground_truth/XX/map_world.npz")["points"]` (или актуальный ключ из вашего map_from_scans).
2. Загрузить позы для seq: прочитать `sequences/XX/poses.txt` (и при необходимости скорректировать через `calib.txt`), получить массив 4×4 матриц, по одной на кадр.
3. Получить список кадров: по файлам в `sequences/XX/velodyne/*.bin` — для каждого файла есть индекс кадра и `scan_id` (имя без расширения).

### Для каждого кадра (pose) в последовательности XX

Для кадра с индексом `i` и `scan_id` (например `000123`):

1. **Поза и карта в СК кадра**
   - Взять матрицу позы для кадра `i`: `T_world_to_scan = inv(poses[i])`.
   - Перевести все точки карты из мировой СК в СК скана:  
     `points_scan = (T_world_to_scan @ [points_world | 1])[:, :3]`.  
   Это и есть «полная сцена» (GT complete) в системе координат этой позы.

2. **Partial (входной скан)**
   - Загрузить скан: `scan = load_bin(sequences/XX/velodyne/{scan_id}.bin)` → xyz (и при необходимости метки из `labels/{scan_id}.label`).
   - Центрировать: `scan_center = scan.mean(axis=0)`, `scan = scan - scan_center`.  
   То же смещение применить к `points_scan`: `complete = points_scan - scan_center`.

3. **Вокселизация**
   - **Partial:** по точкам `scan` (и меткам) построить воксельную сетку с шагом `voxel_size`: уникальные воксели → центры вокселей, метки — мажоритарный голос по вокселю. Получить `partial_coord`, `partial_labels`. В `precompute_voxelized_dataset.py` метки из `.label` приводятся к 20 классам (LEARNING_MAP как в SemanticKITTI), затем мажоритарный голос — так кэш совпадает с датасетом.
   - **Complete:** то же по точкам `complete` (метки нулевые). Получить `complete_coord`, `complete_labels`.

4. **Субсэмплинг**
   - Если точек больше `max_points`: случайно выбрать `max_points` индексов без повторений и оставить только их для partial и отдельно для complete. Так ограничиваем размер каждого сэмпла.

5. **Дополнительные поля**
   - Цвет из высоты (например z → RGB градиент): `partial_color`, `complete_color`.
   - Нормали: можно нулевые или простые, как в текущем датасете: `partial_normal`, `complete_normal`.
   - Сохранить `scan_center` (вектор из 3 float).

6. **Сохранение**
   - Один файл на кадр:  
     `{output_dir}/XX/{scan_id}.npz`  
   - Внутри (массивы numpy):  
     `partial_coord`, `partial_color`, `partial_normal`, `partial_labels`,  
     `complete_coord`, `complete_color`, `complete_normal`, `complete_labels`,  
     `scan_center`.  
   - Рекомендуется `np.savez_compressed(...)` для экономии места.

### Итоговая структура на диске

```
{output_dir}/
  00/
    000000.npz
    000001.npz
    ...
  01/
    ...
  10/
    ...
```

Один `.npz` — один кадр (одна поза, одна seq). Размер одного файла — порядка 0.5–1 MB (сжатый). На 10 train seq (~19k кадров) — примерно **10–20 GB**.

## Когда запускать

- **После** того как map_from_scans уже отработал и в `ground_truth/XX/` лежат `map_world.npz`.
- До обучения диффузии (или перед длительными экспериментами). Запуск один раз, можно в фоне или по ночам.

## Как используется при обучении

- В датасете (SemanticKITTI) при создании передаётся `voxelized_cache_dir={output_dir}`.
- В `__getitem__(idx)` сначала проверяется: есть ли файл кэша для этого `idx` (путь строится по seq и scan_id). Если есть — загружается соответствующий `.npz`, при необходимости применяется лёгкая аугментация (поворот/флип по координатам), данные конвертируются в тензоры и возвращаются. Никакой загрузки map_world и никакой вокселизации на лету.
- Если файла кэша нет — используется прежний «медленный» путь: загрузка map_world, преобразование в СК кадра, вокселизация в `__getitem__`.

Обучение запускается с аргументом, например:  
`--voxelized_cache_dir {data_path}/voxelized_cache`.

---

## Краткая схема пайплайна

1. **map_from_scans** (уже есть)  
   velodyne + poses + calib → одна карта на seq → `ground_truth/XX/map_world.npz`.

2. **Предрасчёт вокселизованного кэша** (эта процедура)  
   Для каждой (seq, кадр): map_world + поза → карта в СК кадра; скан + вокселизация partial/complete; субсэмплинг; сохранение одного маленького `.npz` на кадр.

3. **Обучение**  
   Датасет с `voxelized_cache_dir` только читает готовые `.npz` → быстрая подача батчей, выше загрузка GPU.

Логика и объём данных рассчитаны так, чтобы «тяжёлую» работу (карта в координатах каждой позы + вокселизация) делать один раз, а не на каждом батче.

## Изменения в скрипте прекомпута (прогресс и ускорение)

### Отображение прогресса

- **Полоска (tqdm):** включены `mininterval=1.0` и `miniters=1` — раз в секунду обновляются время работы и ETA, даже если счётчик кадров пока не изменился (один кадр может обрабатываться 30–60+ секунд при большом `map_world`). Так видно, что процесс не завис.
- **Флаг `--verbose`:** при запуске `data/precompute_voxelized_dataset.py` с `--verbose` перед обработкой каждого кадра выводится строка вида  
  `[00] frame 34/4541 000034`  
  так можно видеть, над каким кадром идёт работа.

Пример запуска с подробным выводом:

```bash
python data/precompute_voxelized_dataset.py \
  --data_path /path/to/SemanticKITTI/dataset \
  --output_dir /path/to/SemanticKITTI/dataset/voxelized_cache \
  --voxel_size 0.05 \
  --sequences 00 01 02 03 04 05 06 07 08 09 10 \
  --verbose
```

### Ускорение: вокселизация на GPU

По умолчанию вокселизация выполняется на CPU (numpy). Для ускорения можно использовать **GPU** (PyTorch):

- В скрипте прекомпута добавлен аргумент **`--backend torch`**. При его указании вокселизация облака **complete** (большое облако карты в СК кадра) выполняется на GPU; **partial** (скан с метками) по-прежнему вокселизуется на CPU с мажоритарным голосом по меткам.
- Требуется PyTorch с поддержкой CUDA. На машине без GPU используется CPU.

Пример:

```bash
python data/precompute_voxelized_dataset.py \
  --data_path /path/to/SemanticKITTI/dataset \
  --output_dir /path/to/SemanticKITTI/dataset/voxelized_cache \
  --voxel_size 0.05 \
  --backend torch \
  --sequences 00 01 02 03 04 05 06 07 08 09 10
```

Скрипт `run_voxelized_cache_semantickitti.py` поддерживает `--backend torch` и передаёт его в прекомпут:

```bash
python scripts/run_voxelized_cache_semantickitti.py --data_path /path/to/dataset --backend torch
```

### Почему map_from_scans был быстрым, а прекомпут — медленным

- **map_from_scans** (режим `--save_only_map_world`): для каждой последовательности загружаются сканы по одному (~100k точек), каждый скан переводится в мировую СК (одна матрица 4×4 на маленький облако), все результаты складываются в один массив. Затем **один раз** выполняется вокселизация всей карты на GPU. То есть много лёгких шагов и одна тяжёлая операция на seq.

- **Прекомпут кэша**: для **каждого кадра** нужно перевести **всю** карту `map_world` (десятки миллионов точек на seq) в систему координат этого кадра, затем вокселизовать. То есть на каждый из тысяч кадров делается преобразование огромного облака (на CPU это было узким местом) и вокселизация.

### Что сделано для ускорения прекомпута и загрузки GPU

- **Ограничение числа точек карты (`--max_map_points`, по умолчанию 2M):** как в map_from_scans, мы не используем всю карту целиком. Если в `map_world` больше 2M точек, перед обработкой каждой последовательности карта один раз субсэмплируется до 2M точек (фиксированный seed по seq). Это сильно ускоряет и преобразование, и вокселизацию, и **увеличивает долю времени на GPU** (меньше данных гонять по шине).

- **`--backend torch`**:
  - **Карта на GPU на всю seq:** при `backend=torch` и доступном CUDA карта (уже субсэмплированная) один раз загружается на GPU в виде `(N, 4)` (с единичной последней координатой). Для каждого кадра на GPU выполняются только: загрузка матрицы Tinv (4×4), преобразование `map_homo_gpu @ Tinv.T`, вычитание центра скана, вокселизация на GPU (`voxelize_centers_torch_from_tensor`). Обратно на CPU уходит только результат вокселизации (гораздо меньше точек). Так GPU занят большую часть времени, без лишних пересылок полной карты каждый кадр.
  - Если карта на GPU не используется (например, нет CUDA), при большом числе точек по-прежнему используется преобразование на GPU по чанкам и вокселизация на GPU (с загрузкой данных с CPU каждый кадр).
- При запуске скрипт выводит, используется ли GPU (например: `GPU: NVIDIA A40 (CUDA available, using for transform + voxelize)`). Запускайте с `--backend torch` при наличии CUDA.

## Объём кэша на диске

- Один кадр: один сжатый `.npz` с полями `partial_*`, `complete_*`, `scan_center`. На практике **~0.3–0.5 MB** на кадр (сжатый).
- Одна последовательность (например 00): ~4541 кадр → **~1.5–2.5 GB**.
- Train-сплит (seq 00–07, 09, 10): порядка **~19k кадров** → **~7–15 GB** суммарно в зависимости от плотности сцены и сжатия.

Перед полным прекомпутом убедитесь, что на диске есть хотя бы **15–20 GB** свободного места.

## Идеи из map_from_scans (main) и два сценария

В ветке main в `map_from_scans` есть режим **sliding window**: для каждого кадра берутся только соседние сканы (±25 кадров), в мировой СК склеиваются, вокселизуются один раз, переводятся в СК кадра и сохраняются как **один npz на кадр** — `ground_truth/XX/{scan_id}.npz` с полем `points` (до 200k точек), уже в системе координат скана.

### Что мы из этого используем

1. **Ограничение объёма данных:** в sliding window на кадр идёт не вся карта, а ~51 скан × до 50k точек → порядка 2.5M точек до вокселизации, затем лимит 200k в npz. Мы в прекомпуте сделали то же по смыслу: `--max_map_points` (по умолчанию 2M) при работе от `map_world`.
2. **Быстрый путь по per-frame npz:** если уже запущен `map_from_scans` **без** `--save_only_map_world` (режим sliding window), в `ground_truth/XX/` появляются файлы `000000.npz`, `000001.npz`, … Прекомпут это проверяет: при наличии таких файлов он **не использует map_world** для этих кадров, а читает готовый npz (уже в СК кадра, до 200k точек), центрирует и вокселизует в 0.05. Преобразование десятков миллионов точек с карты не выполняется — только загрузка ~200k и вокселизация. Это сильно быстрее и даёт стабильную загрузку GPU при `--backend torch`.

### Вариант 1 (основной) и опциональный быстрый путь

| Режим | map_from_scans | Прекомпут |
|-------|----------------|-----------|
| **Вариант 1 (основной)** | Один раз с `--save_only_map_world` → только `map_world.npz` | Читает map_world, для каждого кадра переводит в СК кадра (с `--max_map_points` и `--backend torch`), вокселизует, пишет кэш. Минимум места. |
| **Опционально: npz на кадр** | Если ранее запускали map_from_scans без `--save_only_map_world` (sliding window) → есть `ground_truth/XX/{scan_id}.npz` | Для кадров с таким npz прекомпут читает его, центрирует, вокселизует и пишет кэш (без преобразования map_world). Быстрее на кадр, но требует больше места. |

Рекомендуемый поток — только вариант 1: один запуск map_from_scans, затем прекомпут с torch.

### Оценка скорости map_from_scans (sliding window, npz на кадр)

- **На одну seq (например 00, ~4541 кадр):**
  - Загрузка всех сканов в мировую СК и кэш в RAM: порядка 15–25 мин (CPU).
  - На каждый кадр: склейка 51 скана из кэша, вокселизация (например 2–3M точек), перевод в СК кадра, субсэмплинг до 200k, запись npz.
  - С **backend=torch** (вокселизация на GPU): примерно **1–3 с на кадр** → около **2.5–4 ч на seq 00**.
  - С **backend=numpy**: порядка **10–30 с на кадр** → **12–35 ч на seq 00**.

Итого: режим sliding window с созданием npz на каждый кадр при `--backend torch` даёт время на seq того же порядка, что и наш прекомпут с map_world и GPU (после введения `--max_map_points` и карты на GPU). Если сначала запустить map_from_scans в sliding window, а потом прекомпут — прекомпут по этим npz будет проходить заметно быстрее (только чтение + вокселизация без преобразования большой карты).
