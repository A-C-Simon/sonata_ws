# LiDAR-бейзлайн: нужна ли вокселизация по кадрам и как её делать

## 1. Проверка кода бейзлайна с лидаром

Цепочка данных для обучения диффузии по LiDAR:

- **Источники:** `sequences/XX/velodyne/*.bin`, `labels/*.label`, `poses.txt`, `calib.txt`.
- **GT-карта:** `map_from_scans.py` строит одну карту на последовательность → `ground_truth/XX/map_world.npz` (мировая СК).
- **Датасет:** `data/semantickitti.py` — в `__getitem__` для каждого кадра:
  - загружает скан из `.bin`;
  - получает GT: либо из `ground_truth/XX/{scan_id}.npz` (если есть), либо переводит `map_world` в СК кадра через `inv(pose)` (`_load_gt_map_from_map_world`);
  - центрирует по центру скана;
  - **вокселизует** и partial (скан), и complete (GT) в `_voxelize()` с `voxel_size=0.05`;
  - ограничивает число точек до `max_points`, собирает тензоры.
- **Инференс:** `inference.py` — загружает сырой скан, вызывает `prepare_input(scan, voxel_size)`, который тоже вокселизует (grid downsampling), затем прогоняет через модель.

Итого: и при обучении, и при инференсе модель всегда получает **уже вокселизованные** облака (сетка 5 см). Сырые точки в модель не подаются.

---

## 2. Нужно ли вокселизовать данные для каждого кадра из seq?

**Да, по смыслу — для каждого кадра нужна своя пара (partial_voxel, complete_voxel).**

Причины:

1. **Разные позы.**  
   GT «полная сцена» в системе координат кадра получается как `map_world` в мировой СК, переведённый в СК этого кадра: `complete_i = inv(pose_i) @ map_world`. У каждого кадра своя `pose_i`, поэтому `complete_i` разный. Вокселизация применяется уже к этому `complete_i`, значит и результат вокселизации разный для каждого кадра.

2. **Partial тоже свой на кадр.**  
   Кадр = один скан из `velodyne/{scan_id}.bin`. Partial — это вокселизация этого скана. Раз сканы разные, partial-вокселизация тоже своя на кадр.

3. **Модель обучена на вокселах.**  
   Диффузия и refinement работают с точками в сетке 0.05 м (и при обучении, и в инференсе). То есть «вокселизовать по кадрам» нужно не потому, что так устроен датасет, а потому что так устроена модель: на входе и целевом выходе — вокселизованные облака. Вопрос только **где** считать вокселизацию: на лету в датасете или один раз в кэш.

Итого: да, для каждого кадра из seq нужна своя вокселизованная пара (partial, complete). Либо мы считаем её при каждом `__getitem__` (тяжело), либо один раз предрасчитываем и сохраняем в кэш.

---

## 3. Варианты: как это сделать и почему так

### Вариант A: Вокселизация на лету в датасете (как сейчас без кэша)

**Как:** В `SemanticKITTI.__getitem__` без `voxelized_cache_dir`: загрузка скана, загрузка/получение GT (из `map_world` через `inv(pose)` или из per-scan npz), центрирование, `_voxelize(scan)`, `_voxelize(gt_complete)`.

**Плюсы:** Не нужен дополнительный шаг пайплайна, не нужен диск под кэш.  
**Минусы:**  
- При использовании только `map_world.npz` на каждый кадр делается преобразование всей карты в СК кадра (огромный массив × матрица) и две вокселизации. CPU сильно нагружен, GPU часто простаивает, эпохи долгие.

Подходит для отладки и маленьких подмножеств; для полного обучения по 00–10 лучше не использовать.

---

### Вариант B: Предрасчёт вокселизованного кэша (рекомендуется)

**Как:**  
1. После `map_from_scans` с `--save_only_map_world` (одна карта на seq в `ground_truth/XX/map_world.npz`).  
2. Запуск `data/precompute_voxelized_dataset.py`: для каждого кадра (seq, scan_id) по `map_world` и позе получаем complete в СК кадра, вокселизуем partial и complete, субсэмплинг, сохраняем один `.npz` на кадр в `voxelized_cache/XX/{scan_id}.npz`.  
3. Обучение с `--voxelized_cache_dir <path>`: датасет в `__getitem__` сначала проверяет наличие файла кэша; если есть — грузит готовые тензоры (при необходимости с аугментацией), без загрузки `map_world` и без вокселизации.

**Плюсы:**  
- Тяжёлая работа (карта в СК кадра + вокселизация) выполняется один раз.  
- Обучение сильно ускоряется, GPU лучше загружен.  
- Один раз посчитали — много эпох и экспериментов без пересчёта.

**Минусы:**  
- Нужен диск под кэш (~0.5–1 MB на кадр в сжатом виде, порядка 10–20 GB на train 00–10).  
- Нужен один прогон `precompute_voxelized_dataset.py` после `map_from_scans`.

**Почему это самый простой и выгодный вариант:**  
Минимальные изменения в коде (уже добавлены `voxelized_cache_dir` в датасет и флаг в обучение), один скрипт и один проход по данным. Дальше обучение всегда быстрое при тех же данных и том же `voxel_size`.

---

### Вариант C: Per-scan GT без map_world (альтернатива без кэша вокселей)

**Как:** Запуск `map_from_scans` **без** `--save_only_map_world`: в `ground_truth/XX/` пишутся отдельные `{scan_id}.npz` (sliding window по соседним сканам). В датасете тогда в `__getitem__` GT читается из готового npz, без преобразования всей карты. Вокселизация scan и gt по-прежнему на лету.

**Плюсы:** Не нужно переводить гигантский `map_world` на каждый кадр — только загрузка уже готового облака GT и две вокселизации. Быстрее, чем вариант A с `map_world`.  
**Минусы:** Очень много места на диске (сотни GB на все seq), дольше, чем вариант B, и всё равно вокселизация в каждом батче.

Имеет смысл, если по какой-то причине нельзя выделить место под voxelized cache, но можно под per-scan npz.

---

## 4. Что сделано в коде

- В **SemanticKITTI** добавлен параметр **`voxelized_cache_dir`**. Если передан и для данного индекса существует файл кэша, сэмпл берётся из кэша (с опциональной аугментацией), иначе — старый путь (загрузка + map_world/GT + вокселизация).
- В **train_diffusion.py** добавлен аргумент **`--voxelized_cache_dir`** и он передаётся в датасет.
- В **run_lidar_to_sonata.py** пайплайн обновлён:
  - `map_from_scans` вызывается с **`--save_only_map_world`** (чтобы появился `map_world.npz`).
  - Добавлен шаг **precompute_voxelized_dataset** (шаг 1.5).
  - При запуске обучения в **train_diffusion** передаётся **`--voxelized_cache_dir`** на созданный кэш.

Рекомендуемый порядок для LiDAR-бейзлайна:  
`map_from_scans` (save_only_map_world) → **precompute_voxelized_dataset** → обучение с **voxelized_cache_dir**. Так вокселизация по кадрам делается один раз, а обучение идёт максимально быстро.
